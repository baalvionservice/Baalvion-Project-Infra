# Migration Roadmap — Strangler-Fig Extraction

> **No big-bang.** The monolith keeps serving traffic while we peel contexts off
> one at a time, each behind a stable contract. A phase is "done" only when the
> old path is deleted and the new service owns its data, deploy and on-call.

## Guardrails (every phase obeys these)

1. **Contract first** — the gRPC/event contract (`@baalvion/contracts`) ships and
   is consumed *before* any code moves. Producers emit events from the monolith
   first (already done for proxy: `service/domainEvents.js`).
2. **Dual-run, then cut** — new service runs in shadow, reconciled against the
   monolith, before traffic shifts. Feature-flag the cut-over.
3. **Data last** — extract behaviour behind the contract first; split the database
   last (ADR-0003), using logical → physical separation + CDC/outbox backfill.
4. **Reversible** — every cut-over has a documented rollback (flip the flag back).

## Phase 0 — Foundation  ✅ (this prompt)

Shipped: durable event bus (`@baalvion/events` + NATS infra), contracts package
(gRPC + event registry + buf gate), `service-kit`, service catalog + governance
(CODEOWNERS, `baalctl`), Kong gateway, reusable Helm chart, ArgoCD ApplicationSet,
and the **first real event emission** from the proxy monolith. The strangler vine
is planted and load-bearing.

## Phase 1 — Identity extraction (tier-0, highest leverage)

`auth-service` already exists as a separate service. Make it the **single trust
anchor**: every other service verifies tokens via its JWKS / `IdentityService`
gRPC (`service-kit` already wires this). Remove per-service token minting.
- **Exit criteria:** no service mints its own access tokens; Kong + service-kit
  both verify against the identity JWKS; `org.created` consumed downstream.

## Phase 2 — Billing extraction

Stand up `billing-platform` consuming `proxy.session.ended` for usage ingestion
(`BillingService.RecordUsage`, idempotent). The proxy backend stops computing its
own invoices and calls billing instead; proxy keeps only metering.
- **Exit criteria:** invoices generated by billing-platform; proxy emits usage
  events only; `billing.invoice.generated` consumed by notification + audit.

## Phase 3 — Audit + Analytics read models (low-risk, pure consumers)

These are append-only/derived; they have no write coupling, so they extract
cleanly by subscribing to the event stream. Build them as CQRS read models
(audit → signed append-only store; analytics → Timescale/ClickHouse).
- **Exit criteria:** both rebuildable purely from event replay (NATS replay /
  Kafka offset reset) — proven in a DR drill.

## Phase 4 — Trust & Safety + Notifications

Extract `abuse-platform` (already emits `abuse.action.triggered`) and
`notification-platform` (already a service; make it event-driven — it reacts to
`billing.*`, `abuse.*`, `security.incident`, no product sends its own email).

## Phase 5 — Database-per-service split

With behaviour cleanly separated, split the shared Postgres: schema-per-context →
separate instances. Backfill via outbox + CDC; cut reads, then writes, per table.
The append-only audit + analytics stores are already separate (Timescale/CH).

## Phase 6 — Other divisions onto the platform

Commerce, mining, legal, real-estate, jobs adopt `service-kit` + shared identity +
shared billing. They consume platform contracts; they do not reimplement identity,
billing, notifications, or audit.

## Sequencing rationale

```
P0 foundation → P1 identity → P2 billing → P3 audit/analytics ┐
                                                              ├→ P5 DB split → P6 divisions
              P4 trust/notifications ─────────────────────────┘
```
Identity first (everything depends on it). Billing second (revenue path). Pure
consumers (audit/analytics/notifications) in parallel — they're low-risk. DB split
only after behaviour is separated. Other divisions last — they ride the proven rails.
